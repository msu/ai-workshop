Session 5

Discussion & Hands-On Workshop

Student AI policy, Q&A, and pairing

Summary

Open Q&A and discussion on student AI policy: how to teach with agents, how to assess fairly, and how to give students an honest, hopeful path in the AI era.

  • Opening: teach students command line agents honestly -- the session starts with the case for teaching the tools directly and being candid about them.
  • Juniors and the hiring problem -- juniors aren't getting hired because AI does junior work, so the university becomes the safe place to grow junior devs.
  • The tension: compelling output vs fundamentals -- students can generate impressive software, but that may skip the learning we care about.
  • Claude on the Hyperscript parser bug -- it found the bug but proposed three wrong fixes; only parser experience let him steer it right.
  • Reese: letting students use AI -- allowing AI in upper-division courses and making assignments harder in response.
  • A layered assignment approach -- small hand-coded low-level work plus bigger AI-allowed portfolio projects.
  • Locked-down computer labs -- non-networked machines for quizzes as an alternative to written exams.
  • Interview-based grading -- sitting down with students to assess understanding, and why it needs university support.
  • Amazon-style guardrails -- machine-checkable authorization that auto-approves read-only tool calls.
  • Containers for sandboxing -- using Docker-style containers to contain damage, and an agent that hacked out of one.
  • Giving students hope -- the idea that AI replaces everyone is unrealistic; the goal is a balanced, hopeful message.
  • Healthcare AI and data risks -- automating patient transcripts and the HIPAA risk of dumping data into public LLMs.

The Tension

  • The same agent is a patient TA and a tireless cheat
  • It can explain a concept or write the whole assignment
  • The fix is structural, not technological
  good: explain pointers to me
  bad:  write my malloc for me

  same tool, your call

AI and Non-AI Tracks

  • Offer a final project in two tracks
  • Non-AI track is graded for the learning, not the output
  • Let students choose, and make the choice honest
  Track A: no AI
    smaller scope, graded on craft

  Track B: AI allowed
    bigger scope, graded on result

You Have to Write the Code

  • You cannot read code well until you have written it
  • Outsourcing the writing skips the learning
  • The classroom is the safe place to build that skill
  write  -> struggle -> understand
        -> read others' code
        -> build real systems

  skip the first step and the
  rest falls apart

Hands-On

  • Open Claude Code against the course repo
  • Generate a short deck, then a quiz from a notes file
  • Add a CLAUDE.md rule and watch the output change
  > draft a 5-slide deck on
    binary addition

  > generate a 3-question quiz
    from these notes

  > add a rule: code blocks
    max 8 lines

Summary

  • The agent is both a good tutor and an easy cheat
  • The answer is structural: how you assess, not what you ban
  • AI and non-AI tracks let students opt into real work
  • Writing code is the prerequisite for reading code
  • The classroom is the safe place to build that skill
  • Try the tools yourself before you decide your policy
